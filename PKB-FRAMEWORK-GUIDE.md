# Newmedia Jam × PKB Framework 联动指南

> 最后更新：2026-06-17 · 适用：Newmedia Jam v1.5+ × PKB Framework v2.0+

---

## 一、PKB Framework 是什么

PKB Framework 是一个个人知识库管理 skill。它把你的笔记、方案、产出自动整理成可检索的知识库——三层架构（raw → wiki → output），自动生成摘要、建立关联索引、蒸馏你的思维特征。

**对 Jam 用户来说，PKB 的价值很具体：**

- **方案存档**：每次 Jam 产出的会议纪要和草稿包自动存入知识库，不再丢失
- **旧方案检索**：下次做类似选题时，NPC 可以参考上次的方案和经验
- **偏好学习**：积累几次 Jam 后，PKB 会蒸馏出你的策划偏好，让 Jam 的提问和选角越来越精准

**三层架构简图：**

```
PKB 根目录/
├── raw/                ← 你的原始笔记和 AI 产出（Jam 方案存在这里）
│   ├── notes/          ← 个人笔记
│   ├── programs/       ← 方案与项目笔记
│   └── outputs/        ← AI 输出（Jam 的会议纪要和草稿包存在这里）
├── wiki/               ← PKB 自动维护的知识整理层
│   ├── nodes/          ← 笔记摘要（Jam 方案也会生成摘要）
│   ├── mindsets/       ← 你的思维偏好（Jam 读取这里来个性化选角）
│   └── index.md        ← 内容索引
└── CLAUDE.md           ← 项目配置（Claude 平台独有，非必需）
```

**安装 PKB Framework：**

```bash
git clone https://github.com/MoguTown/pkb-skill.git ~/.claude/skills/pkb-framework
```

验证：打开 Claude 输入 `wiki维护`，有反应即安装成功。

> 💡 不装 PKB 也不影响 Jam 全部功能。PKB 是可选增强，不是前置依赖。

## 二、联动做什么

当 Jam 检测到 PKB 已初始化时，自动激活 4 项增强。**不需要任何配置**。

| 增强项 | 什么时候 | 做什么 | 没有PKB时 |
|--------|---------|--------|----------|
| 偏好读取 | Step 1 选题拆解 | 读取你的策划偏好，让提问更精准 | 跳过，使用通用提问 |
| 旧方案检索 | Step 1 + Step 3 | 检索知识库中的过往方案，站在经验之上 | 跳过 |
| 输出归档 | Step 4 纪要 + Step 6 草稿包 | 自动保存到 `raw/outputs/` | 保存到当前工作目录 |
| 存档提示 | Step 6 输出后 | 提示执行 `wiki维护` 归档 | 不提示 |

## 三、PKB 检测机制（跨平台）

Jam 在 Step 1 开始前执行一次 PKB 检测。检测逻辑与 PKB Framework 的 §R.0 根目录定位保持一致，但扩展了确认信号以支持更多平台。

### 检测步骤

```
1. 检查当前工作目录是否含有 raw/ 子目录
2. 如果有，检查是否存在以下任一确认信号：
   - CLAUDE.md          （Claude Code 项目配置）
   - wiki/ 目录         （PKB 初始化后生成）
   - PKB操作指南.md     （PKB 初始化后生成）
   - PKB-dashboard.md   （PKB 初始化后生成）
3. 如果当前目录不满足，向上逐级查找
4. 如果平台不支持文件系统检测，视为"未检测到"
5. 检测成功 → 记录 PKB 根目录路径
6. 仅找到 raw/ 但无确认信号 → 触发温和提示
7. 完全找不到 → 跳过联动
```

### 各平台检测情况

| 平台 | CLAUDE.md | wiki/ | PKB操作指南.md | 说明 |
|------|-----------|-------|---------------|------|
| Claude Code / Desktop | ✅ 有 | ✅ 初始化后有 | ✅ 初始化后有 | 任何信号满足即检测成功 |
| OpenAI Codex | ❌ 无 | ✅ 初始化后有 | ✅ 初始化后有 | 依赖 wiki/ 或 PKB操作指南.md |
| GitHub Copilot | ❌ 无 | ✅ 初始化后有 | ✅ 初始化后有 | 同 Codex |
| 其他平台 | 视情况 | ✅ 初始化后有 | ✅ 初始化后有 | 同上 |

### 关键前提：PKB 必须先初始化

在非 Claude 平台上，**PKB 必须先运行一次 `wiki维护` 完成初始化**，Jam 才能检测到。初始化会生成 `wiki/` 目录和 `PKB操作指南.md`，这些是 Jam 在非 Claude 平台上的主要检测信号。

如果 PKB 未初始化（只有 `raw/` 目录），Jam 会温和提示：
> "我注意到当前目录下有 raw/ 文件夹，看起来像是 PKB 知识库项目，但还没完成初始化。如果你希望方案自动存入知识库，可以先输入 wiki维护 初始化。否则我们直接开始。"

### 一次检测，全程使用

整个会话只在 Step 1 前检测一次。后续步骤直接使用检测结果和记录的根目录路径，不重复检测。

## 四、联动增强的详细执行方式

### 增强A：Step 1 偏好读取

**触发条件**：PKB 已检测到 且 `wiki/mindsets/mindsets.md` 有实质内容

1. 读取 `wiki/mindsets/mindsets.md`
2. 提取策划相关维度（叙事偏好、视觉风格、预算敏感度、载体偏好等）
3. 在概念校准和提问中融入偏好——不是机械引用，而是让提问更精准
4. 在 Step 1 末尾追加偏好确认问题（仅当 mindsets 有实质内容时）

**降级**：mindsets 文件不存在或为空 → 跳过此增强

### 增强B：旧方案检索

**触发条件**：PKB 已检测到

**Step 1**：读取 `wiki/nodes/.tag-index.json` 或扫描 nodes 摘要，检查与本次选题相关的过往方案。如有，在提问中融入。

**Step 3**：当用户主动提及过往项目时，检索对应 nodes 摘要，NPC 自然引用。**不主动检索**——只有用户提及旧项目时才触发。

**降级**：`.tag-index.json` 不存在且 nodes 过多 → 按分类筛选最相关的 5-10 篇摘要

### 增强C：输出归档

**触发条件**：PKB 已检测到

Step 4 会议纪要和 Step 6 产品草稿包的默认保存路径设为 `[PKB根目录]/raw/outputs/`。如果 `raw/outputs/` 不存在则创建，无法创建则回退到当前工作目录。

### 增强D：存档提示

**触发条件**：PKB 已检测到 且草稿包成功存入 `raw/outputs/`

在草稿包末尾追加：`> 💡 方案已存入 PKB 知识库。输入 wiki维护 即可将本次方案归档为可检索的知识资产。`

## 五、设计原则

1. **零侵入**：PKB 联动检测失败不影响 Jam 任何功能
2. **零用户配置**：用户不需要做任何设置，Jam 自动判断
3. **一次检测**：整个会话只在 Step 1 前检测一次
4. **静默增强**：不因为 PKB 联动而增加用户的操作步骤或提问轮次
5. **降级不报错**：任何联动增强失败时静默跳过，不向用户报告技术性错误
6. **跨平台信号**：使用 `PKB操作指南.md` 等初始化产物作为检测信号，不依赖平台特有的 `CLAUDE.md`

## 六、用户操作指南

### 首次使用

1. 安装 PKB Framework（见上方安装命令）
2. 打开你的 AI 工具，**将工作目录设为 PKB 项目根目录**（即含 `raw/` 文件夹的目录）
3. 输入 `wiki维护` 让 PKB 初始化目录结构
4. 在同一对话中说出你的选题，Jam 启动
5. 如果 PKB 检测成功，Step 6 末尾会出现 `💡 方案已存入 PKB 知识库`
6. Jam 结束后输入 `wiki维护` —— PKB 为方案创建摘要、建立关联索引

**从第二次开始直接说出选题即可。**

> ⚠️ **工作目录很重要**：Jam 依靠检测当前目录（及其父目录）来判断 PKB 是否存在。确保你的 AI 工具的工作目录在 PKB 项目内。Claude Code 用户通常已在项目目录下；Codex 用户需注意 `cd` 到正确目录。

### 积累 3-5 次 Jam 后

在 Jam 产出方案 + `wiki维护` 存档后，输入 `mind更新`。

PKB 会蒸馏出你的策划偏好，下次 Jam 时提问更精准、选角向你的偏好倾斜。

### 讨论中提及旧方案

在讨论中直接提即可：

```
"上次做老字号那期，视觉风格定的什么？"
"过去半年我做过几次短视频策划？"
```

Jam 会检索 PKB 知识库中的相关摘要，NPC 可以引用旧方案的经验继续讨论。

### 定期维护

| 时机 | 命令 | 作用 |
|------|------|------|
| 每次 Jam 产出后 | `wiki维护` | 把方案存档到知识库 |
| 积累 3-5 次后 | `mind更新` | 更新策划偏好 |
| 每月 | `lint检查` | 检查知识库健康状况 |

## 七、常见故障

### Jam 产出方案没有出现「已存入 PKB」的提示

**原因**：Jam 没有检测到 PKB 环境。

**按平台排查**：

**Claude 用户**：
1. 工作目录是否在 PKB 项目内？如果不在，`cd` 到 PKB 根目录
2. `CLAUDE.md` 是否存在于项目根目录？

**Codex / 其他平台用户**：
1. 工作目录是否在 PKB 项目内？确保 `cd` 到 PKB 根目录
2. 是否运行过 `wiki维护`？没运行过就没有 `wiki/` 目录和 `PKB操作指南.md`
3. 检查 PKB 根目录下是否有 `wiki/`、`PKB操作指南.md` 或 `PKB-dashboard.md`

**通用排查**：
1. PKB 根目录下有没有 `raw/` 目录？没有就不是 PKB 项目
2. 如果 Jam 提示了"看起来像是 PKB 项目但未初始化"，按提示先运行 `wiki维护`
3. 手动补救：把 Jam 产出文件移到 `raw/outputs/`，再运行 `wiki维护`

### mind更新 后 Jam 选角没有变化

偏好倾斜不是替换——载体匹配优先级始终高于偏好。例如你偏好共情叙事，但选的是短视频载体，小鹿（爆款策划）仍然可能入选。偏好只是辅助权重。

### 两个 Skill 同时安装是否互相干扰

不会。Jam 和 PKB 是独立的 Skill。Jam 只在内部检测 PKB 目录结构，不调用 PKB 的任何功能或修改 PKB 的文件（除了写入 `raw/outputs/`）。PKB 也不会修改 Jam 的任何文件。

### Codex 上 Jam 找不到 PKB

最常见原因：工作目录不在 PKB 项目内。Codex 的 bash 环境与 Claude 不同，确保 `cd /path/to/pkb-project` 后再启动 Jam。如果 PKB 未初始化，先运行 `wiki维护`。

## 八、速查卡

```
每次 Jam 完 → wiki维护
累积 3-5 次 → mind更新
每月        → lint检查
```

| 你要做什么 | 怎么做 | 什么时候 |
|-----------|--------|---------|
| 策划选题 | 直接说选题 | 任何时候 |
| 存档到知识库 | `wiki维护` | 每次 Jam 产出后 |
| 让 Jam 懂你的偏好 | `mind更新` | 累积 3-5 次 Jam 后 |
| 知识库体检 | `lint检查` | 每月一次 |
| 查旧方案 | 讨论中直接提 | Step 3 讨论中随时 |
